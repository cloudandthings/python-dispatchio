# Unified Attempts Design

## Status

Draft proposal for alpha (breaking changes allowed).

## Goals

- Add first-class re-execution of failed jobs.
- Keep the mental model simple for users.
- Make audit history complete and immutable.
- Make completion event correlation deterministic.
- Support default re-execution cascade: failed jobs plus downstream.

## Non-goals

- Backward compatibility with current state schema and event payloads.
- Migration tooling for existing users (none expected in alpha).

## Glossary and naming contract

- `logical_run_id`: cadence-derived run identity (for example day/month/hour key).
- `attempt`: unified execution index for a specific `(job_name, logical_run_id)`, regardless of whether it was scheduler-driven, auto-retry, or manual.
- `dispatchio_attempt_id`: opaque UUID for one concrete submitted attempt.
- `trigger_type`: why an attempt was created (`scheduled`, `auto_retry`, `manual_retry`).
- `trigger_reason`: optional free-text reason for manual/auto-created attempts.
- `trace`: observability metadata map for execution and upstream context.
- `completion_event_trace`: stored copy of raw completion payload used for auditing.

## Type system choice (Enum vs Literal)

Decision for this design:

- Use `Enum` for canonical shared domain fields (for example `status`, `trigger_type`, and `reason` if standardized).
- Use `Literal[...]` for discriminators and local config tags (for example executor `type`, condition `type`, backend `type`).

Why:

- Canonical domain fields are used across orchestrator logic, state, CLI, and tests, and benefit from a single shared type.
- Discriminators are schema tags and are simpler as string literals.

## Core concepts

### Logical run

The cadence-derived run identifier (today/month/hour), currently called `run_id`.

Proposed canonical name in code: `logical_run_id`.

Examples:

- daily: `20260418`
- monthly: `202604`
- hourly: `2026041813`

This is currently referred to as `run_id` across the code, and will be renamed.
Remember, we are in alpha and breaking changes are OK. We dont need to have migrations, deprecations, etc. Just direct changes.

### Attempt number

Unified execution counter for one job and logical run.

- `attempt = 0` is the first scheduled submit.
- `attempt = 1, 2, ...` are later submits, whether automatic or manual.

### Why one unified attempt is simpler

We do not keep separate `rerun` and `attempt` counters.

Instead, we track intent in metadata:

- `trigger_type` captures creation source.
- `trigger_reason` captures operator/orchestrator rationale when relevant.

Simple way to think about it:

- attempt is "the Nth execution of this job for this logical run".
- metadata explains why that attempt exists.

## Correlation model for completion events

### Current issue

Completion assignment today is keyed primarily by `(job_name, run_id)` and can attach stale events to the wrong in-flight execution.

### Proposed rule

Use structured identity fields only.

Completion event must include:

- `job_name`
- `logical_run_id`
- `attempt`
- `dispatchio_attempt_id` (opaque UUID generated by orchestrator on submit)

Correlation priority:

1. Match by `dispatchio_attempt_id` (authoritative).
2. Validate `(job_name, logical_run_id, attempt)` matches stored attempt row.
3. If identity validation fails, do not mutate attempt status; store event in dead-letter/audit table and warn.

Important: no fallback correlation by `(job_name, logical_run_id)` alone.

Trust boundary note:

- `dispatchio_attempt_id` is the authoritative key.
- `trace` is audit/enrichment metadata and not part of correlation.

## Canonical status model

Canonical status values for attempts:

- `submitted`
- `queued` (optional; only when executor can emit it reliably)
- `running`
- `done`
- `error`
- `lost`
- `cancelled`

Notes:

- Keep this set intentionally small.

Supporting fields:

- `reason`: optional string reason for terminal states (used for all terminal states including errors; standardized single field, no separate `error_reason`).

## execution_id format decision

Do not use a parse-dependent combined string as the source of truth.

- Keep `dispatchio_attempt_id` as opaque UUID.
- Keep `job_name`, `logical_run_id`, and `attempt` as separate typed fields.
- If a human-readable combined ID is still useful, treat it as display-only and never parse it for logic.

This avoids delimiter ambiguity (for example if job names contain periods or other separators).

## External executor references and traces

Each executor already keeps a reference/trace object for what it launched. This should be used for audit and observability, not for correlation.

To keep things simple and avoid extra top-level fields, use one `trace` field with a light convention (not a hard schema rule):

- `trace.executor`: executor/runtime trace data (for example process IDs, execution ARN, links).
- `trace.upstream`: upstream trigger/producer trace data when present.

This is a convention only. Additional trace keys are allowed.

Standardized field in state and events:

- `trace: dict[str, Any]`

Examples (always nested under `executor` or `upstream`):

- subprocess/python: `{"executor": {"pid": 1234, "start_time": 1713420000.123}}`
- lambda: `{"executor": {"function_name": "my-fn", "request_id": "..."}}`
- athena: `{"executor": {"query_execution_id": "...", "workgroup": "..."}}`
- step functions: `{"executor": {"execution_arn": "...", "execution_name": "..."}}`
- with upstream context: `{"executor": {"pid": 1234}, "upstream": {"source": "s3", "key": "..."}}`

### Step Functions note

Execution name should not be relied on as unique identity due to 80-char limits and truncation/hash strategies.

- Primary trace identity: `execution_arn`.
- Execution name may include truncation/hash for readability, but correlation should use `dispatchio_attempt_id` + structured identity fields.

Trace update policy:

- `trace` is mergeable metadata and may be updated while an attempt is active.
- Executor-driven updates should primarily patch `trace.executor`.
- Upstream context should primarily be written once into `trace.upstream` at attempt creation.
- Trace updates can be accepted without changing terminal status semantics.
- Correlation remains based only on `dispatchio_attempt_id` and structured identity fields.

## Migration from RunRecord

`RunRecord` is replaced entirely by `AttemptRecord` (the in-memory model for a `run_attempts` row). There is no parallel coexistence.

Field mapping:

| Old `RunRecord` field | New `AttemptRecord` field | Notes |
|---|---|---|
| `job_name` | `job_name` | unchanged |
| `run_id` | `logical_run_id` | renamed |
| `status` | `status` | enum values unchanged except `SKIPPED` removed |
| `attempt` | `attempt` | unchanged; semantics unchanged |
| `submitted_at` | `submitted_at` | unchanged |
| `started_at` | `started_at` | unchanged |
| `completed_at` | `completed_at` | unchanged |
| `error_reason` | `reason` | renamed; used for all terminal states |
| `metadata` | `trace` | renamed; restructure to nested `executor`/`upstream` keys |
| `executor_reference` | `trace.executor` | merged into `trace` under `executor` sub-key |
| _(new)_ | `dispatchio_attempt_id` | new UUID field |
| _(new)_ | `trigger_type` | new enum field |
| _(new)_ | `trigger_reason` | new optional string field |
| _(new)_ | `completion_event_trace` | new optional json field |

The existing `StateBackend` protocol methods that operate on `RunRecord` are replaced by the attempt-aware API described in the State protocol shape section below.

## State model (best audit)

Use immutable per-attempt rows.

### Table: `run_attempts`

One row per submitted attempt.

Columns:

- `dispatchio_attempt_id` (PK, UUID)
- `job_name` (string, indexed)
- `logical_run_id` (string, indexed)
- `attempt` (int, indexed)
- `status` (enum)
- `reason` (string | enum, optional)
- `submitted_at`, `started_at`, `completed_at`
- `trace` (json)
- `completion_event_trace` (json, optional raw event snapshot)
- `trigger_type` (enum: scheduled, auto_retry, manual_retry)
- `trigger_reason` (string, optional)

Constraints:

- unique `(job_name, logical_run_id, attempt)`
- index `(job_name, logical_run_id, attempt desc)`

Allowed state transitions (indicative, not enforced):

- `submitted -> running`
- `submitted -> queued` (optional)
- `queued -> running`
- `submitted -> done`
- `submitted -> error`
- `submitted -> lost`
- `submitted -> cancelled`
- `running -> done`
- `running -> error`
- `running -> lost`
- `running -> cancelled`
- `queued -> error`
- `queued -> lost`
- `queued -> cancelled`

Terminal rows are immutable except for appending `completion_event_trace` and audit metadata.

`trace` may be enriched, or (rarely) updated, over time (for example adding runtime URLs or external IDs) as metadata updates arrive - but metadata attached to previous state records would not be changed.

Idempotency and duplicate completion handling:

- Same `dispatchio_attempt_id` and same terminal status: no-op (idempotent accept).
- Same `dispatchio_attempt_id` and different terminal status: behavior is controlled by `completion_conflict_policy` (see below).

Completion conflict policy (v1):

- `strict` (default): treat conflicting terminal update as `conflict_failure`, do not overwrite, dead-letter and alert.
- `allow_redrive_override` (optional): allow terminal status update when the completion event payload contains `trace.executor.redrive: true`. This field must be explicitly set by the executor's reporter when it detects a redrive (for example, a Step Functions reporter that receives a redriven execution result). Record previous terminal status in audit metadata before overwrite.

Rationale:

- `strict` keeps behavior deterministic and safest by default.
- `allow_redrive_override` prevents operational friction when users redrive executions outside Dispatchio (for example AWS console redrive).

Policy scope:

- `completion_conflict_policy` is configured globally per executor type (for example `sfn: allow_redrive_override`, `subprocess: strict`).
- The global per-executor default can be overridden in project settings/config if needed.
- Rationale: redrive semantics are executor-specific (Step Functions has explicit redrive API; subprocess does not), so executor type is the natural granularity.

### Table: `retry_requests`

Tracks explicit manual re-execution requests for audit.

Columns:

- `id` (PK)
- `requested_at`
- `requested_by` (string)
- `logical_run_id`
- `requested_jobs` (json array; jobs explicitly requested or inferred as initial input)
- `cascade` (bool)
- `reason` (string)
- `selected_jobs` (json array; final job set after applying cascade/no-cascade rules)
- `assigned_attempt_by_job` (json object keyed by job name)

This table is required. Omitting it means manual retry audit history is lost.

## Dependency semantics for attempts

User decision adopted:

- Default re-execution target selection: failed jobs plus downstream closure.
- Option to disable cascade.

Dependency satisfaction rule (latest attempt status only):

- For each dependency job at a given `logical_run_id`, inspect the latest attempt for that dependency job.
- Dependency status is evaluated from that latest attempt only.
- Earlier successful attempts do not satisfy dependencies when a newer attempt exists.

This is intentionally strict and simple: only the latest attempt matters.

## Re-execution selection algorithm (separate, v1)

This section defines only how re-execution targets are chosen. It is independent from retry policy internals.

Algorithm for `dispatchio retry create` with cascade enabled:

1. Resolve requested jobs:
  - If `--jobs` provided, use those jobs.
  - Else, use jobs in terminal failed state (`error` or `lost`) for the specified `logical_run_id`.
2. Build downstream closure from the job dependency graph. All downstream dependencies would need to be rerun when cascading.
3. Union `requested_jobs` and `downstream_jobs`.
4. Dedupe and order deterministically (topological order, then job name for ties).
5. For each selected job, allocate the next attempt number for that `(job_name, logical_run_id)` and schedule it.

Perform all of the above within an atomic database transaction.

When `--no-cascade` is set, only explicitly requested jobs are selected. Clearly warn the user of downstream impact / stale jobs in this case.

## Orchestrator behavior changes

### Scheduling and evaluation

For each job and logical run:

- Evaluate latest attempt for `(job, logical_run_id)`.
- Retry policy creates the next attempt with `trigger_type=auto_retry`.
- Manual re-execution creates the next attempt with `trigger_type=manual_retry`.

Both auto-retry and manual retry use the same atomic attempt allocation: `max(attempt)+1` for the given `(job_name, logical_run_id)`, acquired within a transaction. There is no separate counter per trigger type. A concurrent auto-retry and manual retry for the same job will each get a distinct attempt number; whichever commits first gets the lower number.

### Submission

On submit, always create a new immutable attempt row:

- assign `dispatchio_attempt_id`
- assign `attempt`
- assign `trigger_type` and optional `trigger_reason`
- store `trace` once available

### Completion processing

- Resolve attempt by `dispatchio_attempt_id`.
- Validate structured identity fields as described above.
- Apply `completion_conflict_policy` when a different terminal status is received for an already-terminal attempt.
- Update only that attempt row.

## CLI/API changes

### New CLI commands

- `dispatchio retry create --run-id <logical_run_id> [--jobs ...] [--no-cascade] [--reason ...] [--dry-run]`
- `dispatchio retry list [--run-id ...] [--job ...] [--attempt ...] [--filter attempts|requests] [--limit N] [--offset N]`
- `dispatchio cancel --run-id <logical_run_id> --job <job_name> [--attempt <n>] [--reason ...]`

`retry list` is the single listing command:

- Default view: `attempts` (all execution history).
- Optional `--filter requests`: filter for manual retry request history (`retry_requests`).

### Cancellation semantics

`dispatchio cancel` marks the specified attempt as `cancelled` in the state store.

- If `--attempt` is omitted, cancels the latest active attempt for the given `(job_name, logical_run_id)`.
- Only non-terminal attempts can be cancelled.
- The state store is updated immediately; no signal is sent to the running process.

**Future work:** Propagating a cancellation signal to an in-flight job (process kill, Step Functions stop, etc.) is explicitly out of scope for v1. This will require per-executor cancellation support and should be tracked as a follow-up.

Optional compatibility alias during transition:

- `dispatchio rerun ...` can be supported as alias to `dispatchio retry ...`.

Default `retry create` behavior:

- Find failed jobs for logical run.
- Expand to downstream closure.
- Allocate next attempt number per `(job_name, logical_run_id)`.
- Create attempt rows with `trigger_type=manual_retry` and provided `--reason`.

Also include support for a `--dry-run` flag (default false). When passed, print the selected job set and assigned attempt numbers without writing any state.

Attempt allocation concurrency note:

- Two concurrent retry requests can race on the same `(job_name, logical_run_id)` key.
- Allocation must be atomic per key (for example transactional `max(attempt)+1` with row/key lock), using optimistic concurrency control.

### State protocol shape (breaking)

Replace single-record key semantics with attempt-aware APIs.

Examples:

- `get_latest_attempt(job_name, logical_run_id)`
- `append_attempt(record)`
- `mark_attempt_terminal(dispatchio_attempt_id, ...)`
- `list_attempts(...)`
- `create_manual_retry(...)`

## Logging and audit

### Tick log enrichment

Each action should include:

- `logical_run_id`
- `attempt`
- `dispatchio_attempt_id`
- `trace_ref` (short form)

Note: `trace_ref` is informational only and never used for correlation.
`trace_ref` is a compact display identifier derived from trace (for example execution_arn/query_execution_id/request_id) for operator lookup only; never used in matching or state transitions.

### Auditability outcomes

After this change, operators can answer:

- Which manual retry was requested, by whom, and why.
- Which concrete execution instance produced a terminal status.
- Whether a completion event was accepted or rejected and why.

## Dead-letter handling (minimal viable)

Design principle: simple and effective.

- Always-on internal dead-letter store for application-level rejects.
- Optional external SQS DLQ for AWS transport/consumer failures.
- Alert from internal dead-letter records by default so issues are visible even without external monitoring.

### Why internal dead-letter is required

Many important failures are application-level (for example identity mismatch or invalid state transition) and may happen after a message was successfully received.

These cases should not depend on SQS DLQ behavior.

### Minimal dead-letter reasons (v1)

- `validation_failure`: payload missing required fields or invalid schema.
- `correlation_failure`: `dispatchio_attempt_id` + identity fields do not resolve to a valid attempt.
- `conflict_failure`: duplicate `dispatchio_attempt_id` with incompatible terminal update.
- `policy_reject`: invalid state transition or other orchestration rule violation.

Note:

- Under `completion_conflict_policy=allow_redrive_override`, a conflict that is explicitly identified as redrive is not dead-lettered; it is accepted and audit-annotated.

### Minimal dead-letter record (v1)

Fields:

- `id` (UUID)
- `occurred_at` (datetime)
- `source_backend` (`local` | `aws`)
- `reason_code` (enum/string from list above)
- `reason_detail` (short text)
- `raw_payload` (json)
- `dispatchio_attempt_id` (optional)
- `job_name` (optional)
- `logical_run_id` (optional)
- `attempt` (optional)
- `status` (`open` | `replayed` | `ignored`)

### Runtime behavior (v1)

On rejected completion event:

1. Write dead-letter record to internal store.
2. Emit warning log with `reason_code` and identifiers.
3. Trigger alert via configured alert handler (deduplicated by reason + identifiers for a short window).
4. Continue processing other events (do not fail the whole tick).

### Local vs AWS behavior (v1)

Local mode:

- Use the same configured state backend for dead-letter records.
- Alerts are primary visibility mechanism.

AWS mode:

- Same internal dead-letter write behavior as local.
- Optional: additionally send failed raw message to SQS DLQ if configured.
- Do not require SQS DLQ for correctness.

### Minimal operator surface (v1)

CLI:

- `dispatchio deadletter list [--reason ...] [--status ...] [--job ...] [--limit N] [--offset N]`
- `dispatchio deadletter show <id>`
- `dispatchio deadletter replay <id>`
- `dispatchio deadletter ignore <id> [--note ...]`

Replay semantics:

- Reprocess the stored raw payload through normal validation/correlation flow.
- On success, mark record as `replayed`.
- On failure, keep/open record and update `reason_detail`.

Edge case: if the referenced attempt is already terminal at replay time, the replay will hit `conflict_failure` and re-dead-letter. In this case the operator should use `dispatchio deadletter ignore` to close the record with an audit note rather than replaying.

### Defaults (v1)

- Internal dead-letter handling: enabled, required.
- Dead-letter alerts: enabled.
- AWS SQS DLQ forwarding: optional, disabled by default unless explicitly configured.
- `completion_conflict_policy`: `strict`.

## Timeline example (single attempt concept)

Example for job `load_sales` and `logical_run_id=20260418`:

1. Initial scheduler run creates `attempt=0` with `trigger_type=scheduled`.
2. Attempt 0 fails with transient infrastructure error.
3. Retry policy submits `attempt=1` with `trigger_type=auto_retry`.
4. Attempt 1 fails with data quality error (non-retriable).
5. Operator fixes upstream data and triggers manual retry.
6. Orchestrator creates `attempt=2` with `trigger_type=manual_retry` and `trigger_reason="upstream data corrected"`.
7. Attempt 2 succeeds.

Interpretation:

- A single `attempt` sequence captures all executions.
- Metadata distinguishes scheduler, auto-retry, and manual retry intent.

## Simplicity guardrails

To keep Dispatchio simple:

- Keep one logical run identifier concept for users.
- Use one integer counter (`attempt`) per job and logical run.
- Use opaque UUID for exact execution identity.
- Avoid parsing combined IDs in core logic.
- Default sane behavior (failed + downstream cascade).

## Phased implementation plan

### Phase 1: Data and model primitives

- Add attempt-aware models and immutable attempt store.
- Add `dispatchio_attempt_id` and `trace` fields.

### Phase 2: Correlation hardening

- Upgrade reporters/receivers to CompletionEvent v2.
- Correlate by `dispatchio_attempt_id` and validate structured identity fields.
- Add dead-letter handling for mismatched events.

### Phase 3: Retry orchestration

- Add manual retry creation API and CLI.
- Implement failed + downstream cascade default and `--no-cascade`.
- Wire dependency evaluation against latest attempt status.

### Phase 4: Executor alignment

- Ensure every executor can emit and optionally enrich `trace` for audit purposes.
- Step Functions: prefer ARN-based trace identity; execution name is informational.

### Phase 5: UX and docs

- Add operator docs for retries, attempts, and audit workflows.
- Update examples with attempt inspection and trigger metadata.

## Testing strategy (required)

- Unit tests:
  - attempt numbering across scheduled, auto-retry, and manual retry creation.
  - immutable attempt history.
  - event correlation acceptance/rejection cases.
  - dependency latest-attempt semantics.
  - cascade vs no-cascade retry selection.
- Integration tests:
  - subprocess/python with stale completion events.
  - Step Functions/Athena completion matching by UUID + structured identity fields.
  - Trace enrichment updates during active attempts.
- CLI tests:
  - `retry create`, `retry list` (attempts view + requests view).

## Risks and mitigations

- Risk: payload drift between executors and reporters.
  - Mitigation: shared CompletionEvent v2 model and contract tests.
- Risk: increased query complexity.
  - Mitigation: clear indexes and helper APIs for latest-attempt lookup.
- Risk: operator confusion about why a later attempt exists.
  - Mitigation: always show `trigger_type` and optional `trigger_reason` alongside `attempt`.

## Open design knobs

- None for v1. Any additional options should be tracked in a follow-up design iteration.

## Explicitly out of scope for v1

- Full metrics subsystem and dashboards.
- Partial cascade filtering (beyond on/off cascade).
- Cryptographic signing and verification of completion events.
